# Feature Specification: 文章阅读与写作页面目录与编辑重构

**Feature Branch**: `002-article-toc-editor`  
**Created**: 2026-03-13  
**Status**: Draft  
**Input**: User description: "1.查看文章页面：再右侧增加目录，可联动高亮和点击滚动到对于标题 2. 写文章页面重构：顶部左侧可编辑标题，顶部右侧是【草稿】【发布】功能，下面分左右布局，左边时富文本编辑区域，右侧实时预览区域，富文本采用主流库"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 阅读文章时通过右侧目录快速跳转 (Priority: P1)

作为一名阅读文章的用户，我在浏览文章详情页时，希望在右侧看到根据正文标题自动生成的目录，当前阅读到的段落在目录中会自动高亮，并且我可以点击目录中的任意标题，页面会平滑滚动到对应的位置，以便快速定位和跳转长文内容。

**Why this priority**: 这是文章消费端的核心阅读体验优化，直接影响长文的可读性和信息获取效率，是面向所有阅读用户的基础能力，优先级最高。

**Independent Test**: 仅实现右侧目录生成、联动高亮和点击滚动，不改动写作页，也能独立上线并通过真实阅读行为验证用户是否更容易在长文中定位信息。

**Acceptance Scenarios**:

1. **Given** 文章包含多个不同层级的标题（如一级、二级），**When** 用户打开文章详情页，**Then** 右侧自动生成按层级缩进展示的目录，且目录内容与页面内标题文本一致。
2. **Given** 用户正在向下滚动阅读文章正文，**When** 当前视窗内的标题发生变化，**Then** 右侧目录中对应标题自动高亮显示，且只有一个当前标题处于高亮状态。
3. **Given** 用户在右侧目录点击某一标题项，**When** 触发点击操作，**Then** 页面在可接受的时间内平滑滚动到正文中对应标题位置，且滚动结束时正文标题清晰可见未被顶部遮挡。
4. **Given** 文章中存在非常长的内容或较多标题，**When** 用户多次点击目录任意项，**Then** 页面每次都能稳定、准确地滚动到对应位置，不出现错位或卡顿明显的情况。

### User Story 2 - 写文章时在同一页面完成标题、内容编辑和实时预览 (Priority: P1)

作为一名写作用户，我在撰写或编辑文章时，希望在页面顶部左侧可以直接编辑文章标题，顶部右侧可以一键切换保存为草稿或发布文章，正文区域采用左右布局，左侧为富文本编辑区域，右侧为实时预览区域，以便在同一屏幕内看到排版效果并快速调整内容。

**Why this priority**: 写作端是内容生产入口，重构后的布局与交互直接影响作者创作效率和体验，是与阅读端同等重要的核心流程。

**Independent Test**: 即使不依赖阅读页目录功能，仅上线新的写作布局（标题编辑、草稿/发布、左右编辑+预览），也可以独立完成从写作到保存/发布的完整闭环。

**Acceptance Scenarios**:

1. **Given** 用户进入写文章页面，**When** 页面加载完成，**Then** 顶部左侧显示标题输入区域（支持新建与编辑已有标题），顶部右侧展示清晰可见的“草稿”和“发布”操作入口。
2. **Given** 用户在标题输入区域编辑文本，**When** 用户暂停输入，**Then** 标题内容会在合理频率内自动同步到右侧预览区域，且预览中的标题样式与阅读页保持一致。
3. **Given** 用户在左侧富文本区域进行常见编辑操作（如输入文字、设置粗体/斜体、插入列表或链接等），**When** 完成操作后短时间内，**Then** 右侧预览区域实时更新显示排版效果，且内容与编辑内容一致无明显延迟。
4. **Given** 用户点击顶部右侧“草稿”操作，**When** 保存成功，**Then** 系统以草稿状态保存当前文章内容，并给予明确的成功反馈（如提示文案或状态标识），用户在不离开当前页面的前提下可继续编辑。
5. **Given** 用户点击顶部右侧“发布”操作，**When** 文章满足必要字段校验（如标题非空等），**Then** 系统将文章状态更新为已发布，并给出成功反馈；若校验不通过，则提示具体缺失或错误项而不发布。

### User Story 3 - 作者在写作时通过预览感知阅读侧目录和结构 (Priority: P2)

作为一名写作用户，我在写作页面希望右侧预览不仅展示排版效果，也能大致反映出阅读端的结构（包括标题层级带来的目录结构），从而在创作阶段就可以调整标题层级与内容组织，让读者在阅读页获得清晰的目录体验。

**Why this priority**: 该能力将创作端与阅读端体验串联起来，帮助作者在创作过程中就优化文章结构，减少发布后依赖读者反馈再调整的成本。

**Independent Test**: 即使暂不实现完整阅读页目录，仍可以通过预览中的标题层级展示，独立验证作者对文章结构的感知与调整行为是否得到改善。

**Acceptance Scenarios**:

1. **Given** 用户在写作页左侧编辑标题层级（如将文本设置为一级或二级标题），**When** 右侧预览刷新，**Then** 预览中应准确体现不同层级的视觉层次差异，便于作者识别结构。
2. **Given** 文章包含多级标题，**When** 用户在预览中快速浏览整篇文章，**Then** 可以通过标题样式和层级快速理解整体结构，不需要逐行阅读才能理解文章大纲。

### Edge Cases

- 当文章没有任何标题时，右侧目录区域应如何展示（如隐藏目录区域或显示“暂无目录”文案），以避免出现空白或异常布局。
- 当文章只包含少量标题或标题集中在文章末尾时，目录的显示和联动是否仍保持一致体验。
- 当文章内容极长、标题极多或标题文本过长时，目录中的展示与滚动性能如何保证不会明显影响阅读体验。
- 当用户在写作页频繁编辑标题（新增、删除、修改层级）时，右侧预览的刷新策略应避免明显卡顿或闪烁。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 阅读文章页面右侧必须展示基于正文标题自动生成的目录列表，目录应按标题层级进行缩进或分级展示，保证用户可以从目录中看出文章结构。
- **FR-002**: 系统必须根据用户当前视窗中正在阅读的标题实时更新目录中的高亮项，始终保持只有一个“当前阅读位置”的目录项处于高亮状态。
- **FR-003**: 用户在目录中点击任意标题项时，页面必须在可接受的时间内平滑滚动到正文中对应标题的位置，并确保标题不会被固定头部等元素遮挡。
- **FR-004**: 当文章标题或正文内容发生变更（包括在写作页编辑或更新后），阅读页的目录必须能够基于最新内容重新生成，保证目录与正文始终一致。
- **FR-005**: 写文章页面必须采用顶部功能区 + 下方左右布局的整体结构：顶部左侧为标题编辑区域，顶部右侧为草稿/发布操作区域，下方左侧为富文本编辑区域，下方右侧为实时预览区域。
- **FR-006**: 标题编辑区域必须支持新建文章时输入标题，也支持编辑已有文章时展示并修改原有标题；标题修改后需与文章内容一并保存。
- **FR-007**: 富文本编辑区域必须支持常见文本编辑能力（如段落、标题、多级列表、链接、引用等），并满足主流写作场景下的排版需求；具体采用何种富文本实现方式不在本规格中限定。
- **FR-008**: 预览区域必须在用户编辑内容或标题后，以合理频率自动刷新（而非每次键入都重绘导致性能问题），展示与阅读页接近的排版效果，便于用户预判发布后的呈现。
- **FR-009**: 草稿操作必须允许用户在不满足发布条件的情况下保存当前内容，保存完成后给予明确反馈，并可在后续继续从草稿状态进入编辑。
- **FR-010**: 发布操作必须在关键字段校验通过（如标题非空、正文有内容等）后才允许执行，执行后文章状态变为已发布；若校验不通过，则需给出明确、人类可读的错误提示。
- **FR-011**: 系统必须对草稿与已发布文章在展示上进行明显区分，避免用户误以为草稿已对外公开或误操作发布。
- **FR-012**: 目录生成逻辑必须支持从一级到六级（H1–H6）的所有标题层级，并在目录中全部展示；较低级别标题在目录中通过缩进或样式区分层级，避免信息丢失。
- **FR-013**: 当前版本仅支持“作者”角色，作者可以直接保存草稿和发布文章，无需额外审核或多角色审批流程；未来如需引入编辑/管理员及审核流，将通过新规格单独扩展。
- **FR-014**: 页面必须同时适配桌面端和移动端：桌面端保持右侧目录和预览的左右布局，小屏或移动设备下正文采用单列布局，目录与预览通过折叠菜单或 Tab 切换等形式按需展开，确保主要阅读和写作操作在小屏上也可顺畅完成。

### Key Entities *(include if feature involves data)*

- **文章（Article）**: 表示一篇对外展示的文章，包含标题、正文内容、状态（草稿/已发布等）、创建与更新时间等信息。
- **草稿（Draft）**: 表示尚未对外公开的文章版本，通常与用户账户关联，可以多次保存和继续编辑，最终可转化为已发布文章或覆盖为新内容。
- **目录项（TOC Item）**: 表示文章中某个标题在目录中的一条记录，包含标题文本、层级信息、在正文中的位置标识（用于滚动定位）等，不直接暴露底层实现细节。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在包含多级标题的长文场景下，至少 80% 的真实用户反馈其“找到文章中某个段落的时间”较之前明显缩短（可通过可用性测试或问卷收集），并且从点击目录到页面滚动结束的时间在常规网络环境下不超过 1 秒。
- **SC-002**: 新写作页面上线后，在同类型创作任务下，用户完成“从空白到保存草稿”的平均用时较旧版减少至少 20%，且过程中与布局或找不到功能相关的支持反馈数量下降。
- **SC-003**: 在典型网络和设备环境下，用户进行连续编辑操作（如持续输入、切换标题样式）时，右侧预览的更新不会导致明显卡顿，预览刷新延迟控制在用户可接受范围内（例如大部分操作的可见延迟不超过数百毫秒级别）。
- **SC-004**: 在功能上线后的一段观察期内，与“文章结构不清晰”“找不到对应小节”等相关的用户反馈数量显著下降（例如至少降低 30%），表明目录和结构展示有效提升了阅读体验。
