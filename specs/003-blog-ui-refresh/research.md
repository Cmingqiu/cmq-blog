# Research & Decisions: 优化博客 UI 与发布体验

## 1. 技术栈与约束确认

- **Next.js + App Router**：沿用现有项目架构，在 `src/app` 下完成首页与写作页的 UI 与路由改动，不拆分新应用。
- **UI 技术栈**：继续使用 TailwindCSS + shadcn/ui，所有新增组件（如头像下拉菜单、提示条等）优先基于现有组件或 shadcn/ui 封装。
- **绿色系品牌与宪章冲突**：
  - 宪章要求 UI 以 neutral 为基调，拒绝蓝/绿作为主色调。
  - 本 Feature 需求为设计绿色系 logo 并替换首页文案与 favicon。
  - **决策**：绿色仅用于 logo 图形与 favicon 以及极少量小面积点缀，不作为全局 primary 颜色或大面积背景色；整体布局与组件仍以中性色为主，保证宪章约束不被破坏。

## 2. 发布流与跳转行为

- 现有发布 API 位于 `src/app/api/posts/route.ts`，发布成功后返回新建文章的数据（至少含 id/slug）。
- **决策**：
  - 发布成功后在前端写作页面处理响应：显示短暂“发布成功”提示（若不影响节奏），随后通过路由跳转至首页。
  - 首页列表依赖的文章数据源保持不变，只需保证发布成功后的数据源更新即可：
    - 如果首页直接从数据库读取，无需额外处理，只要跳转即可。
    - 如果有缓存层或静态化，需在实现阶段检查是否需要重新验证或触发 revalidate（视项目现状具体实现）。

## 3. 错误与表单反馈

- 表单校验：
  - 前端：在点击发布前校验标题与正文是否为空，给出明确中文文案。
  - 后端：保持必要的服务端校验，以防绕过前端提交非法数据。
- 接口错误提示：
  - 捕获 fetch/请求异常与非 2xx 响应，解析并转换为用户可理解的错误消息。
  - 建议使用统一的提示组件（如 shadcn/ui 的 toast 或自定义 alert bar），但本 Feature 不强制引入新的全局通知系统，优先复用现有实现。

## 4. 写作页面布局与预览

- 目标：桌面宽屏下编辑区域宽度接近阅读页，预览的排版样式与阅读页相近。
- **决策**：
  - 直接复用或参考阅读页使用的内容容器宽度类（如 `prose` 宽度、最大宽度容器），在写作页中统一。
  - 不引入复杂的双栏布局；以单列编辑或编辑 + 简单预览为主，保持实现简单。
  - 顶部文案替换为返回首页的箭头按钮，使用标准的可点击元素，带可访问性属性（如 aria-label）。

## 5. 首页 UI 与品牌

- 首页精简：
  - 移除线条风格与「Markdown 写作、标签分类、搜索、评论与主题切换」等入口，保留文章列表及必要的导航结构。
  - 确保移除后不会导致路由失联（例如写作入口可以通过其他路径进入，后续可在更合适位置补充）。
- Logo 与 favicon：
  - 绿色系 logo 设计本身可通过简单图形（如首字母变形）或独立文件完成，不在本研究中做具体图形设计。
  - favicon 采用该 logo 的简化版本，尺寸与格式遵循浏览器普遍支持的标准（如 32x32 PNG 或 ICO）。

## 6. 登录头像与下拉菜单

- 用户信息来源：沿用 `src/server/auth` 中现有会话/用户模型，通过已有 guard 或 hook 获取当前用户及头像信息（若无头像则用首字母占位）。
- **决策**：
  - 头像组件放置在首页顶部右侧导航区域，仅在登录状态下显示。
  - 下拉浮层可基于现有 dropdown/menu 组件实现，包含：
    - 「我的文章」：跳转到基于当前用户过滤的文章列表页。
    - 「退出」：调用现有登出 API 或动作，成功后回到未登录视图。

## 7. 决策汇总（符合模板格式）

- **Decision**: 使用 TailwindCSS + shadcn/ui 实现首页与写作页改造，绿色仅用于 logo 与 favicon 点缀，保持整体 UI 中性色调。
  - **Rationale**: 既满足「绿色系 logo」的品牌诉求，又不破坏宪章对 neutral 基调的强制要求。
  - **Alternatives considered**:
    - 全站主色替换为绿色：与宪章冲突，被否决。
    - 放弃绿色系 logo 需求：与业务/品牌诉求冲突，被否决。

- **Decision**: 发布成功后由写作页前端逻辑负责跳转首页，不额外引入发布结果中间页。
  - **Rationale**: 写作体验更流畅，减少额外交互步骤，实现简单。
  - **Alternatives considered**:
    - 留在写作页仅提示成功：缺少从读者视角确认展示的环节。
    - 跳转到文章详情页：不能满足“回首页看整体效果”的需求。

- **Decision**: 使用统一内容容器宽度和 markdown 渲染样式，使写作预览与阅读页排版尽量一致。
  - **Rationale**: 复用现有样式可减少维护成本，并直接提升写作预期与实际展示一致性。
  - **Alternatives considered**:
    - 单独维护一套写作预览样式：增加维护负担，易产生偏差。
