# Feature Specification: 优化博客 UI 与发布体验

**Feature Branch**: `003-blog-ui-refresh`  
**Created**: 2026-03-14  
**Status**: Draft  
**Input**: User description: "1. 发布文章完成后跳转到首页；2. 接口报错和反馈需提示给用户；3. 写文章页面宽度太小，需要最大，【写作预览与阅读页展示一致；发布前请填写标题与正文】替换成返回首页的箭头按钮；5. 去除线条风格，去除【Markdown 写作、标签分类、搜索、评论】；6. 设计绿色系logo , 替换首页的【CMQ Blog】和favicon;7. 首页顶部右侧增加登录入口，如果用户已登录，显示用户头像，鼠标放上有下拉浮层：我的文章和退出"

## User Scenarios & Testing *(mandatory)*



### User Story 1 - 发布文章后返回首页并看到结果 (Priority: P1)

作为已登录的博客作者，我在写作页面完成文章编辑并点击发布后，希望页面自动跳转回首页，在首页文章列表中明显看到刚刚发布的文章，这样可以快速确认发布是否成功并从读者视角检查展示效果。

**Why this priority**: 文章发布与结果确认是博客的核心主流程，跳转和展示反馈直接影响用户对系统可靠性与效率的感知，应优先保障。

**Independent Test**: 只实现该故事时，可通过「登录账号 → 进入写作页 → 填写标题与正文 → 点击发布 → 自动跳转首页 → 在列表中看到新文章」这一完整流程进行端到端验收。

**Acceptance Scenarios**:

1. **Given** 用户已登录且在写作页面填写了合法的标题和正文，**When** 点击发布按钮，**Then** 发布成功、系统自动跳转到首页且新文章出现在文章列表靠前位置。
2. **Given** 用户发布成功后在首页，**When** 刷新首页页面，**Then** 新文章仍然可以在列表中正常展示且链接可点击进入详情页。

---

### User Story 2 - 写作过程中的错误与约束有清晰提示 (Priority: P1)

作为写作者，当发布或保存文章时出现接口错误、网络问题或未填写必填项时，我希望在当前页面得到明确的错误提示或表单校验提示，而不是静默失败或只在控制台报错，这样可以立即知道问题原因并进行修正。

**Why this priority**: 错误与表单反馈是保证数据准确性和提升写作体验的基础能力，如缺失会导致用户误操作和内容丢失风险。

**Independent Test**: 单独实现该故事时，可通过模拟接口异常、网络失败、未填写标题/正文等场景，检查页面是否有可见、易理解的提示信息。

**Acceptance Scenarios**:

1. **Given** 用户在写作页面未填写标题或正文，**When** 点击发布按钮，**Then** 页面在对应字段附近展示必填提示文案并阻止请求发送。
2. **Given** 后端发布接口返回错误（例如 4xx/5xx），**When** 用户点击发布，**Then** 页面在显眼位置显示「发布失败」类错误提示，包含简要原因或重试建议。
3. **Given** 网络中断或超时，**When** 用户点击发布，**Then** 页面提示网络相关错误信息，并允许用户再次重试或返回首页。

---

### User Story 3 - 宽屏写作与预览阅读一致 (Priority: P2)

作为经常在桌面端写作的作者，我希望写作页面的内容区域在大屏下可以最大化宽度，并且预览效果与正式阅读页的版式尽量一致，同时原有「写作预览与阅读页展示一致；发布前请填写标题与正文」提示被替换为返回首页的箭头按钮，这样写作时就能更真实地感知最终呈现效果，并可随时一键返回首页。

**Why this priority**: 写作体验直接影响作者使用频率与满意度，宽度和预览一致性能显著提升长文写作的舒适度，但优先级略低于发布与错误反馈主流程。

**Independent Test**: 实现该故事后，可在不同分辨率下打开写作页，检查编辑区最大宽度、排版是否与阅读页一致，以及顶部返回首页箭头按钮是否始终可用。

**Acceptance Scenarios**:

1. **Given** 用户在桌面端宽屏浏览器中打开写作页面，**When** 浏览器宽度拉到较大，**Then** 主要内容区宽度接近阅读页排版宽度且不会出现明显的「窄列」布局。
2. **Given** 用户在写作页面顶部看到返回首页的箭头按钮，**When** 点击该按钮，**Then** 页面无阻塞地跳转回首页且不会误触发发布。
3. **Given** 用户在写作页面使用预览功能，**When** 比对预览与实际阅读页的同一篇文章，**Then** 段落、标题、列表等基础排版风格基本一致。

---

### User Story 4 - 首页简洁聚焦内容与品牌 (Priority: P2)

作为读者，我进入首页时希望界面更加简洁，去除线条风格和「Markdown 写作、标签分类、搜索、评论」等干扰元素，保留「主题切换」开关，同时看到统一风格的绿色系 logo 和 favicon 替换原「CMQ Blog」文案，这样可以更专注于内容本身并对站点品牌形成清晰记忆。

**Why this priority**: 首页是访问入口，简洁的视觉与一致的品牌识别有利于提升整体感受，但不直接影响基础使用路径，因此设为 P2。

**Independent Test**: 单独发布该故事时，只调整首页视觉和品牌元素即可，通过人工对比修改前后的首页 UI 和浏览器标签图标完成验收。

**Acceptance Scenarios**:

1. **Given** 用户访问首页，**When** 页面加载完成，**Then** 不再出现线条风格装饰及「Markdown 写作、标签分类、搜索、评论」相关入口，「主题切换」开关依然保留。
2. **Given** 用户访问首页，**When** 观察页面顶部和浏览器标签，**Then** 顶部站点标识展示为绿色系 logo（而非纯文字「CMQ Blog」），浏览器标签处显示与之风格一致的 favicon。

---

### User Story 5 - 已登录用户在首页查看个人入口 (Priority: P3)

作为已登录用户，我在首页顶部右侧希望能够看到自己的头像，鼠标悬停或点击头像时出现下拉浮层，包含「我的文章」和「退出」入口，这样可以快速进入个人文章列表或安全退出登录。

**Why this priority**: 该能力对多次使用的活跃用户体验重要，但不会阻塞首次访问和阅读，因此设为 P3。

**Independent Test**: 只实现该故事时，可通过「登录 → 回到首页 → 查看头像与下拉菜单 → 进入我的文章 → 执行退出」流程进行功能验收。

**Acceptance Scenarios**:

1. **Given** 用户已成功登录，**When** 打开或刷新首页，**Then** 顶部右侧展示用户头像（或带首字母的占位头像），未登录状态下不显示头像。
2. **Given** 用户已登录且看到头像，**When** 鼠标悬停或点击头像，**Then** 出现下拉浮层，包含「我的文章」「退出」两个可点击选项。
3. **Given** 用户在下拉浮层中点击「我的文章」，**When** 请求成功，**Then** 页面跳转到只显示当前用户文章的列表视图。
4. **Given** 用户在下拉浮层中点击「退出」，**When** 请求成功，**Then** 当前会话被清除、页面视图切换为未登录状态且头像入口消失。

### Edge Cases

- 发布接口成功但首页数据缓存或列表未及时刷新时，用户在首页未看到刚发布的文章，应如何保证数据同步或给出提示。
- 发布过程中网络波动导致超时重试，多次点击发布可能造成重复提交，应如何去重或防抖。
- 登录状态过期或后端会话失效时，首页仍显示头像但下拉操作失败时的处理方式（自动跳转登录页或清除本地状态后提示重新登录）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统在用户成功发布文章后 MUST 自动跳转到首页，并在首页文章列表中展示新发布文章（包含标题和跳转详情的链接）。
- **FR-002**: 系统在写作页面 MUST 对文章标题和正文作为必填字段进行校验，未填写时阻止提交并在字段附近给予明确中文提示。
- **FR-003**: 系统在调用发布或其他写作相关接口时 MUST 将后端返回的错误或网络异常以用户可见的方式提示在页面上（例如通用错误条或消息提示），而不是只在控制台输出。
- **FR-004**: 写作页面在桌面端 MUST 允许主要内容区域在合理范围内最大化宽度，并使预览区域与正式阅读页在字体、段落宽度和基础排版风格上尽量一致。
- **FR-005**: 写作页面顶部 MUST 使用返回首页的箭头按钮替换原有文案提示「写作预览与阅读页展示一致；发布前请填写标题与正文」，该按钮在任何编辑状态下都可正常返回首页。
- **FR-006**: 首页 MUST 去除现有线条风格装饰，并移除「Markdown 写作、标签分类、搜索、评论」等入口或控件，但 MUST 保留「主题切换」开关。
- **FR-007**: 首页和站点整体 MUST 使用统一的绿色系 logo 作为品牌标识，替换顶部「CMQ Blog」文字，同时浏览器 favicon MUST 更新为与该 logo 一致的绿色系图标。
- **FR-008**: 首页顶部右侧 MUST 提供登录入口。对于已登录用户，展示用户头像，并在悬停或点击时展示包含「我的文章」「退出」操作项的下拉浮层。
- **FR-009**: 系统 MUST 提供「我的文章」视图入口，从首页头像下拉点击后可跳转到仅展示当前登录用户文章的列表页面。
- **FR-010**: 系统在用户选择「退出」后 MUST 结束当前登录会话、清理本地登录状态，并将首页 UI 更新为未登录视图（包括不再展示头像及个人菜单）。

### Key Entities *(include if feature involves data)*

- **文章（Post）**: 代表用户发布的博客文章，包含标题、正文、作者信息、发布时间、在首页列表中的展示信息（包括 slug 或 id 用于跳转详情）。
- **用户（User）**: 代表登录系统的使用者，至少包含唯一标识、昵称或展示名、头像链接（或用于生成占位头像的基础信息）以及登录状态。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 90% 以上的作者在发布文章后 5 秒内能够在首页看到刚发布的文章，并确认跳转行为符合预期（通过可用性测试或问卷统计）。
- **SC-002**: 与发布或保存失败相关的支持反馈（如「点了没反应」「不知道有没有成功」）在发布该功能后 1 个月内较当前基线减少至少 50%。
- **SC-003**: 在桌面端宽屏环境下，至少 80% 的活跃作者在调研中认为「写作区宽度与阅读页接近」且对写作体验给出「满意」或以上评价。
- **SC-004**: 新首页上线后，访问首页的用户中有 80% 以上能够在 3 秒内识别出站点品牌颜色和 logo 样式，且与旧版相比首页主要内容区域的可见面积提升。
